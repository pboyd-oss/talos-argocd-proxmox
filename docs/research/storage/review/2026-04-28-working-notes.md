# Storage Review Working Notes

Date: 2026-04-28

Purpose: preserve review context while evaluating the Talos/Proxmox/Omni storage, PVC backup/restore, ArgoCD sync waves, VolSync/Kopia, Kyverno policy, and pvc-plumber design.

Status: working notes, not final recommendations.

## User Requirements Captured

- Self-hosted Sidero Omni provisioning Talos on Proxmox.
- Declarative Kubernetes/GitOps restore flow.
- No clickops, no GUI restore ritual, no one-off scripts as the normal path.
- Minimal per-app overhead: ideally PVC label only.
- PVC data should restore automatically when a cluster is rebuilt or an app/PVC is recreated.
- If a backup exists, the app must not start and initialize a fresh empty state before restore.
- If no backup exists, a new app should start fresh without manual intervention.
- Serial or slower full restore is acceptable if it preserves correctness.

## First-Pass Findings

### Confirmed Strong Parts

- The design correctly identifies a real gap: VolSync VolumePopulator can restore at PVC creation, but it does not decide "restore if backup exists, otherwise create empty."
- Admission-time mutation is the right timing for this requirement because `dataSourceRef` must exist before the PVC binds.
- The "backup label" interface is clean and low-overhead for app manifests.
- Separating database backup from generic PVC backup is correct. CNPG/Barman is a better fit for Postgres than filesystem-level PVC backup.
- Kyverno performance guardrails are present in the ClusterPolicy:
  - `background: false`
  - `mutateExistingOnPolicyUpdate: false`
  - `generate.synchronize: false`
- The repo already accounts for previous Kyverno blast-radius incidents through namespace exclusions and an emergency webhook cleanup script.
- Kyverno itself is reasonably hardened for homelab scale: 3 admission replicas, topology spread, PDB, and infrastructure namespace exclusions are present.
- Kopia maintenance exists as a CronJob with stable identity to avoid ephemeral VolSync mover identity issues.

### Potentially Important Risks

- The currently included `volsync-pvc-backup-restore.yaml` has `validationFailureAction: Audit` while its docs describe a fail-closed deny gate. Need verify Kyverno semantics for `validate.deny` under this setting, but this looks suspicious.
- The initial repo had newer CEL `ValidatingPolicy`/`MutatingPolicy` files that were not included in `infrastructure/controllers/kyverno/kustomization.yaml`. This has been resolved by deleting those unwired files; the cluster bridge is deliberately the older combined `ClusterPolicy` until the controller migration.
- pvc-plumber `/readyz` is improved from earlier reviews, but current source only checks:
  - Kopia connected at startup.
  - repository path still `stat()`s.
- pvc-plumber `/exists` returns HTTP 200 with `exists:false` and an `error` field when Kopia list or JSON parsing fails.
- Current Kyverno mutation precondition checks only `backupCheck.exists == true`; it does not appear to deny PVC creation when `/exists` returns an error with `exists:false`.
- That means the system may still fail open if `/readyz` passes but the specific `/exists` query fails.
- `HTTP_TIMEOUT` exists in pvc-plumber config but does not appear to be used around Kopia CLI calls in `main.go` or `internal/kopia/client.go`.
- pvc-plumber pre-warms cache with `kopia snapshot list --all --json`; failures are logged and then it falls back to on-demand checks.
- Positive and negative cache entries are cached only when `result.Error == ""`, which is good. Error results are not cached.
- Pre-warmed positive entries expire after `CACHE_TTL`; with deployment `CACHE_TTL=5m`, a full rebuild should only hit pre-warm cache for the first five minutes unless per-request checks refresh entries.
- Initial monitoring gap:
  - pvc-plumber exposed `/metrics`, but no ServiceMonitor/PodMonitor was found.
  - VolSync restore-pending alert tried to identify restore PVCs via `kube_persistentvolumeclaim_info{volumeattributesclass=~".*volsync.*"}`. This was replaced during hardening with a protected-PVC pending alert keyed off the real `backup` label.
- Kopia maintenance currently uses `kopia maintenance run --full --safety=none` from `infrastructure/storage/volsync/kopia-maintenance-cronjob.yaml:140-148`.
- That maintenance job runs at `0 3 * * *`, while hourly VolSync backups generated by Kyverno run at `0 * * * *`. At 03:00 UTC, maintenance can overlap with hourly backup writers.
- Kopia upstream docs warn `--safety=none` disables safety features and requires the operator to ensure no concurrent operations are happening. Given the generated hourly schedule, that guarantee is not currently encoded.

## External Research Findings

- Kyverno current docs say legacy `ClusterPolicy` and `CleanupPolicy` are deprecated as of v1.17, with support planned only through a removal window. Current docs list `ValidatingPolicy`, `MutatingPolicy`, `GeneratingPolicy`, and `DeletingPolicy` under `policies.kyverno.io/v1` as stable. Source: <https://kyverno.io/docs/policy-types/overview/>.
- Kyverno validate docs state `Audit` allows the resource and records a report, while `Enforce` blocks creation/update. They also state `spec.validationFailureAction` is deprecated in favor of rule-level `validate.failureAction`. Source: <https://kyverno.io/docs/policy-types/cluster-policy/validate/>.
- Kyverno CEL policies support `http.Get()` / `http.Post()` from ValidatingPolicy and MutatingPolicy. Source: <https://kyverno.io/docs/policy-types/cel-libraries/>.
- ArgoCD sync waves apply lower waves first and repeat until all phases/waves are in-sync and healthy; the next wave is the first out-of-sync or unhealthy wave. This supports the repo's use of custom health checks as a load-bearing part of ordering. Source: <https://argo-cd.readthedocs.io/en/latest/user-guide/sync-waves/>.
- VolSync VolumePopulator docs say a PVC can reference a ReplicationDestination and VolSync will populate from `status.latestImage`. If no `latestImage` exists, the PVC remains Pending until a snapshot becomes available. Source: <https://volsync.readthedocs.io/en/stable/usage/volume-populator/index.html>.
- Official upstream Backube VolSync still documents rsync/rclone/restic in the main README, while Kopia support appears as an open PR (`backube/volsync#1723`) from `perfectra1n`. This repo uses the perfectra1n fork, not upstream Backube VolSync. Source: <https://github.com/backube/volsync/pull/1723>.
- perfectra1n VolSync docs say Kopia mover supports concurrent access and recommends a single repository for deduplication. Source: <https://perfectra1n.github.io/volsync/usage/kopia/index.html>.
- perfectra1n VolSync docs explicitly recommend one shared Kopia repository for repository-wide deduplication and identity isolation; they state multiple clients can write concurrently. Source: <https://perfectra1n.github.io/volsync/usage/kopia/repository-organization.html>.
- perfectra1n VolSync docs note filesystem repository via `moverVolumes` is available for ReplicationSource only, not ReplicationDestination. Source: <https://perfectra1n.github.io/volsync/usage/kopia/index.html>.
- Kopia official docs support local/network-attached filesystem repositories, and Kopia Repository Server exists to proxy repository access and avoid direct storage credentials on every client. Sources: <https://kopia.io/docs/repositories/> and <https://kopia.io/docs/repository-server/>.
- Kopia official maintenance docs say `--safety=none` disables all safety features and can introduce repository corruption unless the operator guarantees no concurrent operations and repository storage is in sync. Source: <https://kopia.io/docs/advanced/maintenance/>.
- Longhorn official restore remains explicit: select/declare a backup to restore, then create PV/PVC from the restored volume or use CSI snapshot-related objects. No conditional create-time restore decision found. Source: <https://longhorn.io/docs/1.10.1/snapshots-and-backups/backup-and-restore/restore-from-a-backup/>.
- Velero restore remains explicit through a Restore operation/CR and restore workflow. It can restore PVCs via snapshots, filesystem backup, or CSI, but it does not provide "plain PVC create decides restore vs empty." Source: <https://velero.io/docs/v1.18/restore-reference/>.
- KubeStash has Kubernetes-native volume population for PVC restore, but the PVC explicitly references a KubeStash Snapshot via `dataSourceRef`; it does not remove the need to choose a snapshot/source. Source: <https://kubestash.com/docs/v2025.7.31/guides/volume-populator/pvc/>.
- K8up restore is an explicit Restore object/job or CLI-created restore CR. Source: <https://docs.k8up.io/k8up/2.15/how-tos/restore.html>.
- CloudNativePG current recovery docs reinforce the split: Postgres recovery is bootstrapping a new cluster from backup/WAL or snapshots, not generic PVC restore. PITR needs a valid WAL archive. Source: <https://cloudnative-pg.io/docs/1.26/recovery/>.
- Portworx is a useful enterprise comparison point, but current public pages show Enterprise and Backup as 30-day free trials, not a permanent free homelab/community replacement. Their historical "Portworx Essentials" free announcement exists from 2020, but the current pricing page emphasizes trial-to-paid enterprise offerings. Sources: <https://portworx.com/pricing-and-features/> and <https://portworx.com/blog/announcing-portworx-essentials/>.

## Working Judgment

Given the user's target is an enterprise-like managed-platform experience without vendor cost, "unknown backup truth" should block PVC creation.

Reasoning:

- A platform-like storage layer should prefer visible outage over silent empty-state initialization.
- The main threat model is accidental app/PVC deletion, namespace nuking, and rebuild workflows; silent fresh boot is exactly the failure being eliminated.
- New-app first boot is still supported by an authoritative "no backup exists" response.
- Backend/query errors are not the same as "no backup exists"; they are "I cannot prove safety."

Draft rule:

> `exists=true` means restore, `exists=false` with no error means create fresh, and any error/unknown/unreachable state means deny PVC admission.

Second high-confidence safety rule:

> Routine Kopia maintenance should not use `--safety=none` unless the platform also encodes a hard guarantee that no backup/restore/check clients are operating on the repository. The current schedule does not encode that guarantee, so default safety is the better platform default.

## Context Handoff Summary

The user is running Talos on Proxmox with self-hosted Sidero Omni and wants a production/enterprise-like storage/restore experience without paying for Portworx/Kasten-style products. Requirements are no GUI restore, no clickops, no normal-path scripts, declarative GitOps restore, low per-app overhead, and safe automatic PVC restore after cluster/app/PVC deletion.

The current architecture is:

- Longhorn for app-private block PVCs.
- VolSync + Kopia for PVC backup/restore.
- Shared Kopia filesystem repository on TrueNAS NFS.
- Kyverno generates ExternalSecret, ReplicationSource, and ReplicationDestination from PVC label `backup: hourly|daily`.
- Kyverno mutates PVC CREATE with `dataSourceRef` when pvc-plumber says a backup exists.
- pvc-plumber is a custom Go service in `/home/vanillax/programming/pvc-plumber`.
- ArgoCD waves order storage/VolSync before pvc-plumber before Kyverno before apps.

Important user clarification:

- Longhorn's GUI/manual restore model does not scale to ~100 apps and is not acceptable.
- VolSync's VolumePopulator restore pattern is relevant, but by itself it is not enough if it requires the PVC to always know/declare the restore source. The user needs conditional restore intent.
- Portworx is the enterprise benchmark, but current public offering appears trial/paid, not a free permanent fit.

Current review conclusion:

- Keep the custom pvc-plumber/Kyverno restore-intent layer. No researched tool replaces the exact behavior.
- The design is valid because it solves "restore intent" rather than backup mechanics.
- The main hardening need is fail-closed correctness for unknown/error backup checks.

Most important local findings:

- `infrastructure/controllers/kyverno/kustomization.yaml` includes only:
  - `volsync-pvc-backup-restore.yaml`
  - `volsync-nfs-inject.yaml`
  - `volsync-orphan-cleanup.yaml`
- Newer `volsync-pvc-validate.yaml` and `volsync-pvc-mutate.yaml` existed but were not included. They were deleted in the hardening pass to avoid shipping dead code.
- Included `volsync-pvc-backup-restore.yaml` is legacy `ClusterPolicy` and sets `validationFailureAction: Audit`.
- Kyverno docs say Audit records/allows; Enforce blocks. This conflicts with docs claiming fail-closed.
- pvc-plumber `/exists` returns HTTP 200 and `exists:false` with an `error` field on Kopia command/JSON errors.
- Current Kyverno mutation logic checks `backupCheck.exists == true`; it does not deny on `backupCheck.error`.
- Therefore there is likely still a fail-open path: `/readyz` passes, `/exists` query errors, Kyverno treats it as no backup, PVC creates empty.
- Kopia maintenance uses `--safety=none`, and its 03:00 UTC schedule can overlap the generated hourly backup schedule. Upstream Kopia docs warn this can corrupt repositories unless no concurrent operations are happening.
- Explicit PVC inventory currently has 25 backup-labeled PVCs plus 2 backup-labeled StatefulSet templates. The top-of-hour scheduling pattern creates a backup herd; this is manageable at current size but should be monitored and eventually staggered.

Recommended contract:

| pvc-plumber result | Admission behavior |
|---|---|
| `exists=true` | allow PVC and add `dataSourceRef` |
| `exists=false` with no error | allow PVC unchanged for fresh app |
| pvc-plumber unreachable | deny PVC |
| Kopia/NFS/query error | deny PVC |
| malformed response | deny PVC |

Likely next steps:

1. Decide implementation shape for fail-closed:
   - Option A: keep old ClusterPolicy but change to Enforce/rule-level failureAction and add deny rule for `/exists` errors.
   - Option B: migrate admission validate/mutate to `policies.kyverno.io/v1` `ValidatingPolicy`/`MutatingPolicy`, plus add error-deny.
   - Option C: change pvc-plumber `/exists` to return non-2xx on backend errors and make Kyverno deny on missing/invalid response.
2. Add pvc-plumber ServiceMonitor/alerts because it is the custom critical admission oracle.
3. Change Kopia maintenance to default safety and stop forcing daily full GC, or otherwise encode a real no-concurrent-writers window.
4. Add drift/reconciliation check for labeled PVCs missing generated ExternalSecret, ReplicationSource, or ReplicationDestination.
5. Validate live Kyverno behavior with a test PVC while pvc-plumber is down and while `/exists` returns an error.
6. Keep VolSync VolumePopulator as the restore mechanism; pvc-plumber remains the conditional decision layer.

## Implementation Notes From Hardening Pass

Code/manifests were updated after the review to make the current bridge safer before the controller/CRD work:

- `pvc-plumber` now uses a tri-state response contract:
  - `decision=restore`, `authoritative=true`, HTTP 200 when a backup exists.
  - `decision=fresh`, `authoritative=true`, HTTP 200 when no backup exists.
  - `decision=unknown`, `authoritative=false`, HTTP 503 when Kopia/S3/backend truth is unavailable.
- `HTTP_TIMEOUT` is now applied to per-request `/exists` backend calls.
- `pvc-plumber` exposes `pvc_plumber_backup_check_total{backend,decision}` so unknown decisions can alert directly.
- Talos deployment is updated to `ghcr.io/mitchross/pvc-plumber:1.5.1` and explicitly sets `HTTP_TIMEOUT=3s`.
- The active Kyverno `volsync-pvc-backup-restore` policy now uses Enforce semantics and denies backup-labeled PVC CREATE when `/exists` is not authoritative.
- Kyverno `apiCall.default` maps pvc-plumber HTTP failures to `decision=unknown`, `authoritative=false`, so service-call failure also denies rather than relying on implicit behavior.
- The Kyverno mutate rule now requires authoritative `decision=restore`, not just `exists=true`.
- The unwired CEL `volsync-pvc-validate.yaml` and `volsync-pvc-mutate.yaml` files were deleted; the active bridge is intentionally the combined `ClusterPolicy` until controller webhook shadow/enforce mode.
- Kopia maintenance now runs default-safe `kopia maintenance run` at `37 3 * * *`, avoiding recurring `--safety=none` and the top-of-hour backup herd.
- Prometheus now scrapes pvc-plumber and alerts on down/unknown/error conditions.
- The brittle restore-pending alert was replaced with `ProtectedPVCPendingTooLong`, which matches `kube_persistentvolumeclaim_labels{label_backup=~"hourly|daily"}` instead of an unproven `volumeattributesclass` label.
- New infographic docs:
  - `docs/pvc-restore-decision-flow.md`

## Remaining Questions For Controller Design

- Controller migration decision: keep the combined Kyverno `ClusterPolicy` as the live bridge until webhook enforce mode. Do not introduce the deleted CEL `MutatingPolicy`/`ValidatingPolicy` bridge.

- For apps with both PVC state and CNPG databases, is temporal skew acceptable, or do any need an app-specific recovery order/marker?
